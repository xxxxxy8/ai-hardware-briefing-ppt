# AI 硬件简报自动生成工具 (AI Hardware Briefing Generator)

本项目为一个基于 Python 和大语言模型 API 协同设计的自动化 PPT 演示文稿生成系统。系统能够自动读取论文文本，进行本地精准切片预处理，并调度 DeepSeek 结构化输出能力，最终利用 python-pptx 渲染出符合考核大纲要求的 3 页高密度学术答辩演示文稿。

 一、 代码运行环境与依赖库

- 操作系统: Windows 11
- Python 版本: Python 3.14
- 核心依赖库:
  - openai (v1.0+ 标准，用于调度兼容集成的 DeepSeek-Chat 模型 API)
  - python-pptx (用于底层高级幻灯片元素的自动化布局与动态渲染)
- 安装命令:
  pip install openai python-pptx

 二、 Prompt 设计思路

为了严格卡死考核大纲要求的 3 页内容并保证生成的专业度，Prompt 采用了“结构化脚手架（Schema Scaffolding）”的设计策略：
1. 角色注入：将模型定义为专业的 PPT 结构生成器，限定其只能以纯 JSON 数组形式返回。
2. 硬性数量约束：在 System Instruction 中下达死命令，强制限制 JSON 数组长度必须且只能为 3。
3. 精准骨架注入：在 Prompt 中直接下发了各页面的标题模板与项数期望（例如：Slide 2 必须且只能包含 3 项，分别对应 Hardware, Algorithm, Application 三层架构），从源头上规避了 AI 自作聪明多生成或遗漏核心要点的问题。
4. 字数流控限制：限制单项 Bullet Point 长度在 12-15 单词以内，防止长句在自动化渲染时导致严重的文本溢出或重叠。

 三、 遇到的主要问题及解决方案

在项目的测试与交付调整过程中，主要经历了以下三次针对生成效果与排版的迭代优化：

1. 问题一：大模型内容生成项数超标（第二页多出 Cross-Layer 项）
   - 现象：在测试运行中，大模型在生成第二页“三层抽象架构”时自由发挥，自发延伸输出了第四项内容（Cross-layer co-design），不符合考核大纲严格限定三层的硬性要求。
   - 解决方案：优化了 API 的 Prompt 提示词，在 Slide 2 的生成规则中加入带有 CRITICAL 标签的死命令（MUST contain EXACTLY 3 bullet points. No more, no less.），同时将大模型的 temperature 降低，弱化其发散能力，从而精准锁死 3 项输出。

2. 问题二：高密度文本导致大模型输出偶发性截断
   - 现象：由于在 Prompt 中对字数、项数和格式做出了极为严苛的限制，导致大模型在处理内容密集的第三页（趋势与挑战）时，偶尔因为格式计算冲突而输出空值，反映在幻灯片上表现为第三页内容留白。
   - 解决方案：在 System 提示词中直接嵌入标准的 JSON 格式 Schema 骨架示例作为容错脚手架，理顺模型的输出逻辑，确保了 100% 的高成功率稳定输出。

3. 问题三：字号偏小，影响阅读
   - 现象：初始生成的标题和正文字体在视觉上略显单薄。
   - 解决方案：微调了 build_presentation 渲染引擎中的排版参数。将标题字号提升至 Pt(36)，正文字号整体放大至 Pt(20)；为了防止字号放大后行间距过大破坏美感，同步微调了段后距（space_after）的自适应分配逻辑，使大字号下的全屏视觉效果更加饱满大气。
